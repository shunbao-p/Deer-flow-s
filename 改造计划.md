# DeerFlow 改造计划

本文档记录基于 DeerFlow 项目的三个系统级改造方向，按照 A → B → C 的顺序逐步推进。
每个方向说明了当前项目的不足、为什么选择这个方向入手、以及具体要做什么。

---

## 改造方向 A：技能自动生成系统

### 当前项目的不足

DeerFlow 的技能系统（Skills）本质上是一批"给模型看的工作说明书"，存放在 `skills/public/` 和 `skills/custom/` 目录下，每个技能的核心是一个手写的 `SKILL.md` 文件。

这种机制存在一个明显的缺陷：

- 用户如果想让 Agent 掌握一个新的工作流，必须自己手写 `SKILL.md`
- `SKILL.md` 需要遵守特定的 YAML frontmatter 格式和内容结构
- 这对普通用户来说门槛极高，几乎等于"只有开发者才能扩展技能"
- 项目目前没有任何方式让用户通过对话或界面来定义新技能

换句话说：**技能系统的扩展能力被锁死在了配置文件层面，而不是用户层面。**

### 为什么选择这个方向

1. **解决了真实存在的问题**：让普通用户也能扩展 Agent 能力，是 Agent 系统走向实用的关键一步
2. **技术覆盖面广**：涉及 Prompt 工程（引导模型生成结构化内容）+ Gateway 接口设计 + 技能系统改造 + 前端交互
3. **演示效果直观**：用户输入几句话，系统自动生成一个可用的技能，效果可以直接录屏展示
4. **贴近当前 AI 热点**：让 Agent "自我学习" 新能力，是 Agent 系统设计的前沿方向之一

### 要做的事情

```
目标：用户通过自然语言对话描述一个工作流，系统自动生成 SKILL.md 并保存

核心流程：
  用户描述工作流
       │
       ▼
  后端引导式对话（收集关键信息：步骤、工具需求、适用场景）
       │
       ▼
  调用 LLM，根据对话内容生成符合格式的 SKILL.md 文本
       │
       ▼
  保存到 skills/custom/ 目录
       │
       ▼
  下次主代理加载时，新技能自动生效
```

**需要新建/修改的文件：**

| 文件 | 类型 | 作用 |
|------|------|------|
| `backend/src/skills/generator.py` | 新建 | 核心生成逻辑：接收对话内容，调用 LLM，输出 SKILL.md 文本 |
| `backend/src/gateway/routers/skill_generator.py` | 新建 | 新增 POST 接口，接收用户输入，触发生成流程 |
| `backend/src/gateway/app.py` | 修改 | 注册新路由 |
| `frontend/src/...` | 修改 | 在技能管理页增加"教学新技能"入口和引导对话 UI |

**简历叙述：**
> 实现了技能自学习系统，用户通过自然语言对话即可定义新的 Agent 工作流，系统自动提炼生成结构化 SKILL.md 配置并持久化，无需手动编写，降低了技能扩展门槛。

## 改造方向 B：Agent 执行过程可视化

### 当前项目的不足

DeerFlow 目前的用户体验是一个标准的聊天界面：用户发送消息，等待，Agent 回复。

在这个等待过程中，用户对 Agent 内部发生的一切一无所知：

- 不知道 Agent 正在执行哪个中间件
- 不知道 Agent 调用了哪些工具、调用顺序是什么
- 不知道 sandbox 里执行了什么命令
- 不知道 memory 是否在这轮对话后被更新
- 一旦 Agent 出错或卡住，用户完全无法判断问题出在哪里

这是一个典型的**黑箱问题**：Agent 的执行过程对用户和开发者都不透明。

从工程角度看，缺乏可观测性（Observability）会导致：
- 调试困难：Agent 行为异常时无从排查
- 信任问题：用户不知道 Agent 在做什么，难以建立信任
- 性能盲区：无法判断哪个环节耗时过长

### 为什么选择这个方向

1. **可观测性是 AI 工程的核心议题**：当前工业界对 Agent 系统的可观测性高度关注，这个方向贴近真实工程需求
2. **全栈改动**：涉及后端中间件事件发送 + SSE 事件流 + 前端实时 UI 组件，展示面广
3. **演示效果最炸裂**：对话时能实时看到执行轨迹，视觉冲击力强，适合录屏和现场演示
4. **改造完 A 之后进行 B，正好对中间件链路已经比较熟悉**，降低上手难度

### 要做的事情

```
目标：在聊天界面右侧或底部增加"执行轨迹面板"
      实时展示 Agent 当前在做什么

效果示意：
  用户发送消息后，面板实时出现：
  ┌─────────────────────────────────┐
  │ ▶ [ThreadDataMiddleware] 初始化线程目录     ✅ │
  │ ▶ [SandboxMiddleware] 准备沙盒环境         ✅ │
  │ ▶ [工具调用] web_search("最新AI论文")      ⏳ │
  │ ▶ [工具调用] bash("python analysis.py")      │
  │ ▶ [MemoryMiddleware] 更新记忆中...           │
  └─────────────────────────────────┘
```

**技术实现思路：**

```
各中间件执行时 → 发出自定义 SSE 事件
工具被调用时  → 发出工具调用事件
sandbox 执行时 → 发出命令执行事件
        │
        ▼（通过已有的 LangGraph 事件流机制）
前端 useThreadStream() 接收新事件类型
        │
        ▼
新建执行轨迹组件，实时渲染事件列表
```

**需要新建/修改的文件：**

| 文件 | 类型 | 作用 |
|------|------|------|
| `backend/src/agents/middlewares/*.py` | 修改 | 各中间件在关键节点发出执行事件 |
| `backend/src/sandbox/tools.py` | 修改 | 工具调用时发出工具事件 |
| `frontend/src/core/threads/hooks.ts` | 修改 | 新增事件类型的接收和处理逻辑 |
| `frontend/src/components/.../execution-trace.tsx` | 新建 | 实时执行轨迹 UI 组件 |

**简历叙述：**
> 为 Agent 系统增加了执行过程透明化机制，通过 SSE 事件流实时向前端推送中间件状态、工具调用轨迹和沙盒执行日志，将黑箱 Agent 改造为可观测系统，提升了可调试性和用户信任感。

## 改造方向 C：向量化记忆系统

### 当前项目的不足

DeerFlow 的 memory 系统已经具备了基础的跨会话记忆能力：每次对话结束后，系统通过 LLM 提取关键信息，存入 `memory.json`，下次对话时注入系统提示词。

但这套系统在**记忆检索**环节存在明显的局限：

- 当前做法是把记忆按固定结构截取，注入前 N 条 facts 进 prompt
- 截取依据是 `confidence` 分数和时间顺序，与当前对话内容**无关**
- 用户说"帮我规划一次旅行"，系统可能把"用户喜欢写代码"这条记忆注入进去，而把"用户去年去过日本"这条漏掉
- 随着记忆累积，这个问题会越来越明显：大量记忆存在，但注入的不是最相关的

核心问题：**记忆存储和检索是两套完全割裂的逻辑，检索不考虑当前上下文的语义相关性。**

### 为什么选择这个方向

1. **解决了 memory 系统真实存在的瓶颈**：语义检索 vs 简单截取，效果差异在长期使用中非常明显
2. **引入向量数据库是核心 AI 工程技能**：ChromaDB、FAISS 等向量数据库在 RAG 系统中广泛使用，是面试高频考察点
3. **RAG 思想的直接应用**：把记忆系统理解为一种 RAG（检索增强生成），展示了对 AI 工程架构的深度理解
4. **改动集中在后端 memory 模块**：代码改动范围明确，风险可控，前端基本不需要改动
5. **完成 A 和 B 之后进行 C，对整个项目架构已经非常熟悉**，可以更从容地处理深层改动

### 要做的事情

```
目标：将 memory 检索从"按 confidence 截取前 N 条"
      升级为"按当前对话语义相关性检索最匹配的记忆"

现在的流程：
  memory.json → 取 confidence 最高的前 15 条 → 注入 prompt

改造后的流程：
  用户当前输入 → 向量化（Embedding）
       │
       ▼
  从向量数据库中检索语义最相近的记忆
       │
       ▼
  只注入最相关的记忆（而不是时序前 N 条）
       │
       ▼
  Agent 的记忆召回更精准
```

**技术实现思路：**

```
存储侧（updater.py）：
  每次提取出新 facts 后
  → 对每条 fact 调用 Embedding 模型获取向量
  → 把向量和 fact 存入 ChromaDB

检索侧（prompt.py）：
  每次对话开始时
  → 对当前用户输入做 Embedding
  → 在 ChromaDB 中做向量相似度检索
  → 取 Top-K 最相关记忆注入 prompt
  → 替换原来的简单截取逻辑
```

**需要新建/修改的文件：**

| 文件 | 类型 | 作用 |
|------|------|------|
| `backend/src/agents/memory/vector_store.py` | 新建 | ChromaDB 封装层：初始化、写入、查询 |
| `backend/src/agents/memory/updater.py` | 修改 | 存储 fact 时同步写入向量库 |
| `backend/src/agents/memory/prompt.py` | 修改 | 检索时改为向量语义查询替代简单截取 |
| `backend/pyproject.toml` | 修改 | 添加 chromadb 依赖 |

**简历叙述：**
> 将 Agent 记忆系统从平铺 JSON 检索升级为向量数据���语义检索，引入 ChromaDB 存储记忆向量，使 Agent 能按当前对话上下文的语义相关性精准召回历史信息，而非简单时序截取，显著提升了长期记忆的有效利用率。

## 整体改造叙述（简历用）

> **基于 DeerFlow 的 AI Agent 系统能力增强**
>
> 在开源 LangGraph Agent 框架基础上进行三个维度的系统级改造：
>
> 1. **Agent 自学习能力（方向 A）**：实现技能自动生成系统，用户通过自然语言对话即可定义新工作流，系统自动提炼生成结构化技能配置并持久化，无需手动编写。
>
> 2. **系统可观测性（方向 B）**：为 Agent 执行过程增加实时追踪机制，通过 SSE 事件流将中间件状态、工具调用轨迹和沙盒执行日志实时推送前端展示，将黑箱 Agent 改造为可观测系统。
>
> 3. **记忆系统升级（方向 C）**：将 Agent 记忆从平铺 JSON 截取升级为向量数据库语义检索，引入 ChromaDB，使 Agent 按上下文语义相关性精准召回历史信息。

## 推进顺序与阶段目标

```
阶段 1：方向 A（技能自动生成）
  目标：做出可演示的 MVP
  完成标志：用户能通过对话生成一个可用的 SKILL.md 并在下次对话中生效

阶段 2：方向 B（执行可视化）
  目标：在聊天界面实时展示 Agent 执行轨迹
  完成标志：发送一条消息后能看到工具调用和中间件状态的实时展示

阶段 3：方向 C（向量化记忆）
  目标：记忆检索改为语义相关性召回
  完成标志：对话中可以验证与当前主题相关的记忆被优先注入，无关记忆被过滤

最低交付线：完成阶段 1 + 阶段 2，已具备较强的简历说服力
理想目标：三个阶段全部完成
```
