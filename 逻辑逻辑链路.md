## 问题 1：前端是从哪个组件提交消息的？

### 先看什么

- [frontend/src/app/workspace/chats/[thread_id]/page.tsx](/mnt/d/Agent项目/deer-flow/frontend/src/app/workspace/chats/[thread_id]/page.tsx)
- [frontend/src/core/threads/hooks.ts](/mnt/d/Agent项目/deer-flow/frontend/src/core/threads/hooks.ts)
- [frontend/src/components/workspace/input-box.tsx](/mnt/d/Agent项目/deer-flow/frontend/src/components/workspace/input-box.tsx)

### 你要理解的核心点

用户在页面里输入内容，不是直接由页面自己发请求，而是经过一层封装：

1. 页面组件负责把 `threadId`、当前上下文、提交函数拼起来
2. 输入框组件负责收集用户输入
3. 真正提交消息的逻辑在 `useSubmitThread(...)` 这个 hook 里

### 代码里的逻辑链

从 [page.tsx](/mnt/d/Agent项目/deer-flow/frontend/src/app/workspace/chats/[thread_id]/page.tsx) 开始看：

- 页面创建了 `handleSubmit`
- `handleSubmit` 是通过 `useSubmitThread(...)` 得到的
- 然后传给了 `InputBox`

也就是说：

`页面 ChatPage -> useSubmitThread -> InputBox -> 用户点击发送 -> handleSubmit`

在 [hooks.ts](/mnt/d/Agent项目/deer-flow/frontend/src/core/threads/hooks.ts) 里，`useSubmitThread` 做了两件关键事情：

1. 如果有附件，先上传文件
2. 再调用 `thread.submit(...)` 发起真正的对话请求

### 这一部分你最终要能说出来的话

你应该能自己复述出这句话：

“用户在 `InputBox` 里输入内容，页面把提交动作交给 `useSubmitThread`，最后由 `thread.submit(...)` 把消息发出去。”

如果这句话你能独立讲出来，说明第一个问题已经过关。

## 问题 2：请求是发到 `/api/langgraph` 还是 `/api/...`？

### 先看什么

- [frontend/src/core/api/api-client.ts](/mnt/d/Agent项目/deer-flow/frontend/src/core/api/api-client.ts)
- [frontend/src/core/config/index.ts](/mnt/d/Agent项目/deer-flow/frontend/src/core/config/index.ts)
- [docker/nginx/nginx.local.conf](/mnt/d/Agent项目/deer-flow/docker/nginx/nginx.local.conf)

### 你要理解的核心点

DeerFlow 有两类后端请求：

1. 主对话请求：走 `/api/langgraph/*`
2. 管理类请求：走 `/api/*`

主对话请求是“Agent 真正在干活”的那条线。

比如：

- 聊天消息
- 流式输出
- thread 状态

这些都走 `/api/langgraph/*`

而下面这些走 `/api/*`：

- 模型列表
- skills 配置
- mcp 配置
- memory 查询
- uploads
- artifacts

### 代码里的逻辑链

在 [api-client.ts](/mnt/d/Agent项目/deer-flow/frontend/src/core/api/api-client.ts) 中，LangGraph SDK 客户端是这样初始化的：

- `apiUrl: getLangGraphBaseURL()`

再去看 [index.ts](/mnt/d/Agent项目/deer-flow/frontend/src/core/config/index.ts)：

- 默认会把 LangGraph 请求指向 `/api/langgraph`

然后看 Nginx 配置 [nginx.local.conf](/mnt/d/Agent项目/deer-flow/docker/nginx/nginx.local.conf)：

- `/api/langgraph/` 会被转发到 LangGraph 服务
- `/api/models`、`/api/memory`、`/api/skills` 等会被转发到 Gateway

### 这一部分你最终要能说出来的话

你应该能自己复述出这句话：

“聊天主链路走 `/api/langgraph/*`，配置和管理类接口走 `/api/*`，Nginx 按路径把它们分流到不同服务。”

## 问题 3：后端哪个入口先接住请求？

### 先看什么

- [backend/langgraph.json](/mnt/d/Agent项目/deer-flow/backend/langgraph.json)
- [backend/src/agents/lead_agent/agent.py](/mnt/d/Agent项目/deer-flow/backend/src/agents/lead_agent/agent.py)
- [backend/src/agents/thread_state.py](/mnt/d/Agent项目/deer-flow/backend/src/agents/thread_state.py)

### 你要理解的核心点

当请求走到 LangGraph 服务后，真正的入口不是 `FastAPI`，而是 LangGraph 配置文件里声明的图。

在 [langgraph.json](/mnt/d/Agent项目/deer-flow/backend/langgraph.json) 里：

- `lead_agent` 指向 `src.agents:make_lead_agent`

这说明主对话请求最终会进入：

- `make_lead_agent(config)`

也就是 [agent.py](/mnt/d/Agent项目/deer-flow/backend/src/agents/lead_agent/agent.py)

### 代码里的逻辑链

你看 `make_lead_agent(config)` 时，要抓住 4 个关键点：

1. 选模型
2. 组装工具
3. 组装中间件
4. 组装系统提示词

最后这些东西一起被 `create_agent(...)` 组装成主代理。

`ThreadState` 在这个阶段也很重要，因为它决定了这个代理运行时能带着哪些状态：

- messages
- sandbox
- thread_data
- title
- artifacts
- todos

这些定义在 [thread_state.py](/mnt/d/Agent项目/deer-flow/backend/src/agents/thread_state.py)

### 这一部分你最终要能说出来的话

你应该能自己复述出这句话：

“主对话请求进入 LangGraph 后，会根据 `langgraph.json` 找到 `make_lead_agent`，然后组装出 DeerFlow 的主代理实例。”

## 问题 4：主代理加载了哪些中间件和工具？

### 先看什么

- [backend/src/agents/lead_agent/agent.py](/mnt/d/Agent项目/deer-flow/backend/src/agents/lead_agent/agent.py)
- [backend/src/tools/tools.py](/mnt/d/Agent项目/deer-flow/backend/src/tools/tools.py)
- [backend/src/sandbox/tools.py](/mnt/d/Agent项目/deer-flow/backend/src/sandbox/tools.py)
- [backend/src/sandbox/sandbox.py](/mnt/d/Agent项目/deer-flow/backend/src/sandbox/sandbox.py)

### 你要理解的核心点

DeerFlow 的主代理不是“模型收到问题后直接生成答案”，而是：

1. 中间件先为它准备上下文和执行环境
2. 工具系统决定它可以做哪些事
3. 模型在这些能力之上再进行推理

这就是 DeerFlow 和普通聊天机器人的根本区别。

### 中间件你重点理解什么

在 [agent.py](/mnt/d/Agent项目/deer-flow/backend/src/agents/lead_agent/agent.py) 里的 `_build_middlewares(...)`，你重点看这几个：

- `ThreadDataMiddleware`
- `UploadsMiddleware`
- `SandboxMiddleware`
- `TitleMiddleware`
- `MemoryMiddleware`
- `ClarificationMiddleware`

这里最重要的理解不是把每个中间件定义背下来，而是知道：

“主代理在真正回答之前，系统已经先帮它准备好了线程目录、上传文件、沙箱环境、记忆上下文等运行条件。”

### 工具你重点理解什么

在 [backend/src/tools/tools.py](/mnt/d/Agent项目/deer-flow/backend/src/tools/tools.py) 中，`get_available_tools(...)` 会把工具汇总起来。

工具来源有几类：

- config.yaml 里配置的工具
- 内置工具
- MCP 工具
- 条件工具，比如 `view_image`
- 子代理工具 `task`

### sandbox 工具你重点理解什么

在 [backend/src/sandbox/tools.py](/mnt/d/Agent项目/deer-flow/backend/src/sandbox/tools.py) 中，Agent 真正可以执行：

- `bash`
- `ls`
- `read_file`
- `write_file`
- `str_replace`

而 [backend/src/sandbox/sandbox.py](/mnt/d/Agent项目/deer-flow/backend/src/sandbox/sandbox.py) 定义了 sandbox 的抽象能力：

- 执行命令
- 读文件
- 写文件
- 列目录

你可以把这一层理解成：

“模型并不会直接改系统文件，它必须通过 sandbox 工具层去做事。”

### 这一部分你最终要能说出来的话

你应该能自己复述出这句话：

“主代理先通过中间件获得运行环境，再通过工具系统获得执行能力，最后模型才能在这些能力基础上完成任务。”

## 问题 5：最终结果是怎么回到前端页面里的？

### 先看什么

- [frontend/src/core/threads/hooks.ts](/mnt/d/Agent项目/deer-flow/frontend/src/core/threads/hooks.ts)
- [frontend/src/components/workspace/messages/message-list.tsx](/mnt/d/Agent项目/deer-flow/frontend/src/components/workspace/messages/message-list.tsx)
- [frontend/src/components/workspace/messages/message-list-item.tsx](/mnt/d/Agent项目/deer-flow/frontend/src/components/workspace/messages/message-list-item.tsx)
- [frontend/src/app/workspace/chats/[thread_id]/page.tsx](/mnt/d/Agent项目/deer-flow/frontend/src/app/workspace/chats/[thread_id]/page.tsx)

### 你要理解的核心点

DeerFlow 前端不是“请求结束后一次性拿完整结果”，而是流式接收。

所以你看到的现象是：

- AI 一边生成
- 前端一边显示
- 如果有 subagent 状态，也会边执行边展示

### 代码里的逻辑链

在 [hooks.ts](/mnt/d/Agent项目/deer-flow/frontend/src/core/threads/hooks.ts) 里：

- `useThreadStream(...)` 负责建立和维护这条流
- `onCustomEvent(...)` 负责接住像 `task_running` 这样的自定义事件
- `onFinish(...)` 负责在流结束后更新最终状态

然后在 [message-list.tsx](/mnt/d/Agent项目/deer-flow/frontend/src/components/workspace/messages/message-list.tsx) 里：

- 消息会被按类型分组渲染
- 普通消息、澄清消息、present-files、subagent 消息各有不同显示逻辑

最后由 [message-list-item.tsx](/mnt/d/Agent项目/deer-flow/frontend/src/components/workspace/messages/message-list-item.tsx) 把单条消息真正渲染到页面。

### 这一部分你最终要能说出来的话

你应该能自己复述出这句话：

“后端结果不是一次性返回，而是通过流持续发给前端，前端在 `useThreadStream` 里接收，然后通过消息列表组件实时渲染。”

## 专题 1：对话主链路

### 目标

彻底搞懂一条普通消息是怎么被处理的。

### 重点文件

- [frontend/src/core/threads/hooks.ts](/mnt/d/Agent项目/deer-flow/frontend/src/core/threads/hooks.ts)
- [backend/langgraph.json](/mnt/d/Agent项目/deer-flow/backend/langgraph.json)
- [backend/src/agents/lead_agent/agent.py](/mnt/d/Agent项目/deer-flow/backend/src/agents/lead_agent/agent.py)
- [backend/src/agents/lead_agent/prompt.py](/mnt/d/Agent项目/deer-flow/backend/src/agents/lead_agent/prompt.py)

### 这个专题到底在解决什么问题

这个专题解决的是：

“用户输入一句话之后，DeerFlow 到底是怎么把它交给主代理处理的？”

如果这个专题你没学透，后面看 sandbox、skills、subagents 时会一直不知道它们插在主链路的哪个位置。

### 真实链路

请你把这条链路按下面 9 步去看，每一步都要真的打开对应文件：

1. 用户在聊天页输入内容  
起点是 [frontend/src/app/workspace/chats/[thread_id]/page.tsx](/mnt/d/Agent项目/deer-flow/frontend/src/app/workspace/chats/[thread_id]/page.tsx)。  
这个页面把 `threadId`、设置项、提交函数组装起来，再交给 `InputBox`。

2. 输入框触发提交  
你再看 [frontend/src/components/workspace/input-box.tsx](/mnt/d/Agent项目/deer-flow/frontend/src/components/workspace/input-box.tsx)。  
这里不是自己直接发请求，而是调用从页面传进来的 `onSubmit`。

3. 真正发请求的是 `useSubmitThread(...)`  
打开 [frontend/src/core/threads/hooks.ts](/mnt/d/Agent项目/deer-flow/frontend/src/core/threads/hooks.ts)。  
这里会先处理附件上传，然后调用 `thread.submit(...)`。这一行就是“把消息送进 LangGraph”的真正动作。

4. `thread.submit(...)` 不是发到 Gateway，而是发到 LangGraph  
继续看 [frontend/src/core/api/api-client.ts](/mnt/d/Agent项目/deer-flow/frontend/src/core/api/api-client.ts) 和 [frontend/src/core/config/index.ts](/mnt/d/Agent项目/deer-flow/frontend/src/core/config/index.ts)。  
这里能看到 LangGraph SDK 客户端的地址默认是 `/api/langgraph`。

5. Nginx 把 `/api/langgraph/*` 转发给 LangGraph 服务  
打开 [docker/nginx/nginx.local.conf](/mnt/d/Agent项目/deer-flow/docker/nginx/nginx.local.conf)。  
你会看到 `/api/langgraph/` 这一段 location，把聊天主请求分流到了 LangGraph，而不是 Gateway。

6. LangGraph 进入哪个 agent，由 `langgraph.json` 决定  
打开 [backend/langgraph.json](/mnt/d/Agent项目/deer-flow/backend/langgraph.json)。  
这里定义了图名 `lead_agent` 对应 `src.agents:make_lead_agent`，所以聊天请求最终会进入主代理工厂函数。

7. 主代理在 `make_lead_agent(...)` 里被组装出来  
打开 [backend/src/agents/lead_agent/agent.py](/mnt/d/Agent项目/deer-flow/backend/src/agents/lead_agent/agent.py)。  
这里做了四件最关键的事：选模型、加载工具、构建中间件、拼系统提示词。  
看到这里时，你要明确：DeerFlow 的主代理不是单个类，而是运行时动态组装出来的。

8. 主代理行为被系统提示词进一步约束  
打开 [backend/src/agents/lead_agent/prompt.py](/mnt/d/Agent项目/deer-flow/backend/src/agents/lead_agent/prompt.py)。  
这里决定了代理的工作方式，比如先澄清再行动、何时使用 subagent、如何处理 working directory、如何引用 skills。

9. 执行结果再流回前端显示  
回到 [frontend/src/core/threads/hooks.ts](/mnt/d/Agent项目/deer-flow/frontend/src/core/threads/hooks.ts) 里的 `useThreadStream(...)`。  
它负责接收流式结果。然后 [frontend/src/components/workspace/messages/message-list.tsx](/mnt/d/Agent项目/deer-flow/frontend/src/components/workspace/messages/message-list.tsx) 和 [frontend/src/components/workspace/messages/message-list-item.tsx](/mnt/d/Agent项目/deer-flow/frontend/src/components/workspace/messages/message-list-item.tsx) 负责把消息渲染出来。

### 这个专题你要重点盯住的连接点

你看这个专题，不要只盯“谁调用谁”，要盯下面 4 个连接点：

1. 前端把什么上下文传给后端  
比如 model、thinking_enabled、plan_mode、subagent_enabled

2. `langgraph.json` 如何把服务入口指向主代理  
这是对话主线真正进入后端的地方

3. `make_lead_agent(...)` 如何把整个运行时拼起来  
这里是 DeerFlow 主干里的“组装中心”

4. `prompt.py` 如何改变代理行为  
DeerFlow 不只是“调用模型”，而是用一套很强的系统提示词去约束模型如何思考、澄清、拆任务、用技能

### 你在这个专题里的阅读顺序

1. 先看 [frontend/src/core/threads/hooks.ts](/mnt/d/Agent项目/deer-flow/frontend/src/core/threads/hooks.ts)  
理解消息是怎么提交和接收流式结果的

2. 再看 [backend/langgraph.json](/mnt/d/Agent项目/deer-flow/backend/langgraph.json)  
理解主代理入口是怎么声明的

3. 再看 [backend/src/agents/lead_agent/agent.py](/mnt/d/Agent项目/deer-flow/backend/src/agents/lead_agent/agent.py)  
理解模型、工具、中间件、prompt 是怎么组装的

4. 最后看 [backend/src/agents/lead_agent/prompt.py](/mnt/d/Agent项目/deer-flow/backend/src/agents/lead_agent/prompt.py)  
理解主代理为什么会表现出现在这种“先澄清、再行动、必要时拆子任务”的行为

### 这个专题学完以后你应该能回答

- 主代理是怎么被创建出来的
- 系统提示词怎么拼出来
- `thinking`、`plan mode`、`subagent mode` 是怎么影响运行逻辑的
- 一条普通消息为什么能流式返回到页面

### 你最终要能自己讲出来的话

“DeerFlow 的对话不是直接把一句话丢给模型，而是前端把消息发给 LangGraph，LangGraph 根据配置找到主代理工厂，主代理再把模型、工具、中间件和系统提示词组装好后运行。”

## 专题 2：sandbox 与文件系统

### 目标

搞懂 Agent 为什么能执行命令、读文件、写文件。

### 重点文件

- [backend/src/sandbox/sandbox.py](/mnt/d/Agent项目/deer-flow/backend/src/sandbox/sandbox.py)
- [backend/src/sandbox/tools.py](/mnt/d/Agent项目/deer-flow/backend/src/sandbox/tools.py)
- [backend/src/sandbox/middleware.py](/mnt/d/Agent项目/deer-flow/backend/src/sandbox/middleware.py)
- [backend/src/config/paths.py](/mnt/d/Agent项目/deer-flow/backend/src/config/paths.py)
- [backend/src/sandbox/local/local_sandbox.py](/mnt/d/Agent项目/deer-flow/backend/src/sandbox/local/local_sandbox.py)

### 这个专题到底在解决什么问题

这个专题解决的是：

“Agent 为什么能像在一台小电脑里一样执行命令、操作文件，而且还不会把不同会话搞混？”

这其实是 DeerFlow 很核心的能力层。没有 sandbox，DeerFlow 就更像普通工具调用聊天，而不是可执行任务的 agent harness。

### 真实链路

请你按一个最简单的例子去理解，比如用户要求“读取一个文件”或者“执行一个 bash 命令”：

1. 主代理已经在对话主链路中运行起来了  
当模型判断自己需要动文件或命令时，它不会直接执行系统调用，而是调用工具，比如 `bash` 或 `read_file`。

2. 工具调用先进入 [backend/src/sandbox/tools.py](/mnt/d/Agent项目/deer-flow/backend/src/sandbox/tools.py)  
这里是 Agent 和 sandbox 之间的桥。  
你把它理解成“工具适配层”最合适：上面接模型，下面接真正的 sandbox。

3. 工具层先确认当前 thread 有没有 sandbox  
还是在 [backend/src/sandbox/tools.py](/mnt/d/Agent项目/deer-flow/backend/src/sandbox/tools.py)。  
这里最重要的不是某一行函数名，而是逻辑：如果当前会话还没有准备好 sandbox，就在这里懒初始化。

4. sandbox 的创建或绑定与中间件有关  
打开 [backend/src/sandbox/middleware.py](/mnt/d/Agent项目/deer-flow/backend/src/sandbox/middleware.py)。  
这个中间件的职责是：让 agent 运行时带着 sandbox 状态。  
也就是说，主代理执行工具之前，系统已经给它预留了“拿到 sandbox”的入口。

5. thread 隔离目录不是凭空来的，而是由 `Paths` 统一管理  
打开 [backend/src/config/paths.py](/mnt/d/Agent项目/deer-flow/backend/src/config/paths.py)。  
这里定义了每个 thread 对应三块目录：
- workspace
- uploads
- outputs

这一步你一定要看明白，因为它解释了为什么同一个系统里多个会话不会写到同一个目录里。

6. 工具层会把虚拟路径转成真实路径  
还是看 [backend/src/sandbox/tools.py](/mnt/d/Agent项目/deer-flow/backend/src/sandbox/tools.py)。  
DeerFlow 让模型统一使用 `/mnt/user-data/...` 这种虚拟路径，是为了让模型不用关心宿主机实际目录结构。  
然后系统再把这些路径翻译成当前 thread 的真实目录。

7. 真正执行命令或文件读写的是具体 sandbox 实现  
先看接口 [backend/src/sandbox/sandbox.py](/mnt/d/Agent项目/deer-flow/backend/src/sandbox/sandbox.py)。  
它只定义能力，不写实现。  
然后再看 [backend/src/sandbox/local/local_sandbox.py](/mnt/d/Agent项目/deer-flow/backend/src/sandbox/local/local_sandbox.py)。  
这里你会看到，本地 sandbox 最终是怎么运行命令、怎么读写文件、怎么做路径转换的。

8. 执行结果再往回走  
具体 sandbox 实现把结果返回给工具层，工具层把字符串结果返回给主代理，主代理再决定如何继续推理，最后再回到前端。

9. 你在脑中要形成的完整句子是  
“模型调用工具 -> 工具层接手 -> sandbox 准备好环境 -> 虚拟路径映射 -> 具体 sandbox 执行 -> 结果返回主代理”

### 你必须看懂的 4 个概念

1. `Sandbox` 抽象接口  
在 [backend/src/sandbox/sandbox.py](/mnt/d/Agent项目/deer-flow/backend/src/sandbox/sandbox.py)  
它定义了“沙箱应该能做什么”，而不是“具体怎么做”。

2. `tools.py` 工具层  
在 [backend/src/sandbox/tools.py](/mnt/d/Agent项目/deer-flow/backend/src/sandbox/tools.py)  
它把 Agent 的工具调用转换成 sandbox 能理解的操作。

3. `Paths` 路径映射  
在 [backend/src/config/paths.py](/mnt/d/Agent项目/deer-flow/backend/src/config/paths.py)  
它负责把 thread 的工作区、上传区、输出区组织起来。

4. 本地实现与容器实现  
本地实现你先看 [backend/src/sandbox/local/local_sandbox.py](/mnt/d/Agent项目/deer-flow/backend/src/sandbox/local/local_sandbox.py)  
它会让你清楚“命令最终是怎么被执行的”。

### 这个专题最重要的理解

你在这里不要只学“能运行 bash”。真正要学懂的是：

- 为什么要有 `/mnt/user-data/workspace`
- 为什么要按 thread 隔离目录
- 为什么 Agent 使用的是“虚拟路径”而不是直接拿系统真实路径
- 为什么同样叫 sandbox，本地模式和容器模式背后实现可以不同

### 你在这个专题里的阅读顺序

1. 先看 [backend/src/sandbox/sandbox.py](/mnt/d/Agent项目/deer-flow/backend/src/sandbox/sandbox.py)  
建立抽象层概念

2. 再看 [backend/src/config/paths.py](/mnt/d/Agent项目/deer-flow/backend/src/config/paths.py)  
理解 thread 隔离目录结构

3. 再看 [backend/src/sandbox/tools.py](/mnt/d/Agent项目/deer-flow/backend/src/sandbox/tools.py)  
理解 Agent 工具调用是怎么进入 sandbox 的

4. 再看 [backend/src/sandbox/middleware.py](/mnt/d/Agent项目/deer-flow/backend/src/sandbox/middleware.py)  
理解 sandbox 何时初始化

5. 最后看 [backend/src/sandbox/local/local_sandbox.py](/mnt/d/Agent项目/deer-flow/backend/src/sandbox/local/local_sandbox.py)  
理解一个具体实现类如何执行命令和路径转换

### 这个专题学完以后你应该能回答

- sandbox 到底是什么
- thread 隔离是怎么做的
- `/mnt/user-data/...` 为什么能映射到真实目录
- 本地 sandbox 和 aio/container sandbox 的区别是什么
- Agent 为什么不能直接碰任意系统路径

### 你最终要能自己讲出来的话

“DeerFlow 不是让模型直接操作宿主系统，而是让模型通过 sandbox 工具层操作每个 thread 的隔离工作区，虚拟路径再映射到真实目录。”

## 专题 3：Gateway API

### 目标

搞懂为什么项目里除了 LangGraph 还要有一个 Gateway。

### 重点文件

- [backend/src/gateway/app.py](/mnt/d/Agent项目/deer-flow/backend/src/gateway/app.py)
- [backend/src/gateway/routers/models.py](/mnt/d/Agent项目/deer-flow/backend/src/gateway/routers/models.py)
- [backend/src/gateway/routers/uploads.py](/mnt/d/Agent项目/deer-flow/backend/src/gateway/routers/uploads.py)
- [backend/src/gateway/routers/artifacts.py](/mnt/d/Agent项目/deer-flow/backend/src/gateway/routers/artifacts.py)
- [backend/src/gateway/routers/skills.py](/mnt/d/Agent项目/deer-flow/backend/src/gateway/routers/skills.py)

### 这个专题到底在解决什么问题

这个专题解决的是：

“为什么 DeerFlow 不把所有接口都塞进 LangGraph 主服务，而要单独放一个 Gateway？”

你可以把 Gateway 看成“外围业务接口层”。它不负责主对话推理，但负责所有前端需要的资源型、管理型接口。

### 真实链路

你要用“上传文件”这个例子来学 Gateway，最容易懂：

1. 前端要上传文件时，不会走 `/api/langgraph/*`  
而是走 [frontend/src/core/uploads/api.ts](/mnt/d/Agent项目/deer-flow/frontend/src/core/uploads/api.ts)。  
这个文件里会请求 `/api/threads/{threadId}/uploads`。

2. Nginx 看到这是 `/api/threads/.../uploads`  
打开 [docker/nginx/nginx.local.conf](/mnt/d/Agent项目/deer-flow/docker/nginx/nginx.local.conf)。  
这里会把这类请求转发给 Gateway，而不是 LangGraph。

3. Gateway 的总入口是 [backend/src/gateway/app.py](/mnt/d/Agent项目/deer-flow/backend/src/gateway/app.py)  
这个文件做的事情很简单但很关键：创建 FastAPI 应用，并把 models、mcp、memory、skills、uploads、artifacts 这些 router 全部挂上去。

4. 上传请求最终进入 [backend/src/gateway/routers/uploads.py](/mnt/d/Agent项目/deer-flow/backend/src/gateway/routers/uploads.py)  
这里负责：
- 接收文件
- 校验文件名
- 保存到 thread 对应的 uploads 目录
- 对 PDF/Word/Excel/PPT 做 markdown 转换
- 返回给前端可访问的路径信息

5. 如果前端之后想显示这个文件，走的不是 LangGraph，而是 artifact 接口  
你再看 [backend/src/gateway/routers/artifacts.py](/mnt/d/Agent项目/deer-flow/backend/src/gateway/routers/artifacts.py)。  
这个 router 会根据 thread_id 和虚拟路径找到真实文件，然后返回文本、HTML 或二进制内容。

6. 同样的模式也适用于其他 Gateway 能力  
- 模型列表请求会进入 [backend/src/gateway/routers/models.py](/mnt/d/Agent项目/deer-flow/backend/src/gateway/routers/models.py)
- skills 管理会进入 [backend/src/gateway/routers/skills.py](/mnt/d/Agent项目/deer-flow/backend/src/gateway/routers/skills.py)
- memory 查询会进入 [backend/src/gateway/routers/memory.py](/mnt/d/Agent项目/deer-flow/backend/src/gateway/routers/memory.py)

7. 所以你可以把 Gateway 的真实链路统一记成  
“前端发资源类请求 -> Nginx 分流到 Gateway -> app.py 找到对应 router -> router 读配置/读文件/写文件/返回 JSON 或文件”

8. 这里最重要的理解是  
Gateway 处理的是“资源型请求”，不是“推理型请求”。

### 这个专题你要盯住的边界

最关键的是把下面这句话想明白：

“凡是偏资源管理、配置管理、文件服务的能力，适合放 Gateway；凡是偏会话推理、工具调用、子代理执行的能力，适合放 LangGraph 主服务。”

这就是 DeerFlow 设计上的职责分离。

### 你在这个专题里的阅读顺序

1. 先看 [backend/src/gateway/app.py](/mnt/d/Agent项目/deer-flow/backend/src/gateway/app.py)  
理解 Gateway 是一个单独的 FastAPI 应用，以及 router 是怎么挂进去的

2. 再看 [backend/src/gateway/routers/models.py](/mnt/d/Agent项目/deer-flow/backend/src/gateway/routers/models.py)  
理解一个最简单的只读接口

3. 再看 [backend/src/gateway/routers/uploads.py](/mnt/d/Agent项目/deer-flow/backend/src/gateway/routers/uploads.py)  
理解文件上传为什么走 Gateway

4. 再看 [backend/src/gateway/routers/artifacts.py](/mnt/d/Agent项目/deer-flow/backend/src/gateway/routers/artifacts.py)  
理解生成文件是怎么被前端访问到的

5. 再看 [backend/src/gateway/routers/skills.py](/mnt/d/Agent项目/deer-flow/backend/src/gateway/routers/skills.py)  
理解为什么技能启用/禁用状态也走 Gateway

### 这个专题学完以后你应该能回答

- Gateway 和 LangGraph 的职责边界
- 为什么上传、技能、模型配置不直接塞进主对话服务
- 前端哪些请求走 Gateway
- 为什么前端需要一个“资源服务层”

### 你最终要能自己讲出来的话

“Gateway 是 DeerFlow 的资源与管理接口层，负责前端所需的模型、技能、memory、上传和产物接口；LangGraph 则负责真正的对话推理和 agent 执行。”

## 专题 4：skills

### 目标

搞懂 DeerFlow 为什么能按需加载某些“能力说明书”。

### 重点文件

- [backend/src/skills/loader.py](/mnt/d/Agent项目/deer-flow/backend/src/skills/loader.py)
- [backend/src/skills/parser.py](/mnt/d/Agent项目/deer-flow/backend/src/skills/parser.py)
- [backend/src/gateway/routers/skills.py](/mnt/d/Agent项目/deer-flow/backend/src/gateway/routers/skills.py)
- [backend/src/agents/lead_agent/prompt.py](/mnt/d/Agent项目/deer-flow/backend/src/agents/lead_agent/prompt.py)

### 这个专题到底在解决什么问题

这个专题解决的是：

“DeerFlow 为什么能像装能力包一样，在不同任务中注入不同工作流说明？”

skills 不是 Python 工具实现本身，它更像“给模型看的工作说明书”。

### 真实链路

请你用“系统启动后，主代理要知道有哪些 skill 可用”这个过程去理解：

1. skill 文件不在 `backend/src` 里，而是在项目根目录下的 `skills/`  
你先打开 `deer-flow/skills` 目录，确认里面有 `public`、`custom` 这样的分类目录。

2. 每个 skill 的入口文件都是 `SKILL.md`  
模型真正要读的不是 Python 代码，而是这份 Markdown 说明。

3. 系统扫描技能目录时会进入 [backend/src/skills/loader.py](/mnt/d/Agent项目/deer-flow/backend/src/skills/loader.py)  
这个文件会做三件事：
- 找到 skills 根目录
- 扫描 `public` 和 `custom`
- 找每个子目录里的 `SKILL.md`

4. 找到 `SKILL.md` 之后，解析工作交给 [backend/src/skills/parser.py](/mnt/d/Agent项目/deer-flow/backend/src/skills/parser.py)  
这里会把 frontmatter 里的信息解析出来，比如 skill 名称、描述、分类。

5. skill 不是找到就一定可用  
`loader.py` 还会去读 extensions 配置，判断这个 skill 当前是否启用。  
所以这里的真实链路是：  
“扫描到 skill -> 解析 skill -> 再检查启用状态”

6. 当主代理构建系统提示词时，会把启用中的 skill 拼进 prompt  
这一段你回到 [backend/src/agents/lead_agent/prompt.py](/mnt/d/Agent项目/deer-flow/backend/src/agents/lead_agent/prompt.py) 看。  
它会把 skills 生成一个提示词片段，然后加入主代理的 system prompt。

7. 到这里 skill 才真正影响模型  
也就是说，skill 不是“被执行”，而是“被注入到提示词后，让模型按里面的流程去做事”。

8. 如果前端要启用、禁用或安装 skill  
这时走的链路是：前端 -> Gateway -> [backend/src/gateway/routers/skills.py](/mnt/d/Agent项目/deer-flow/backend/src/gateway/routers/skills.py) -> 修改扩展配置 -> 下次主代理加载 skills 时生效。

9. 你在脑中要形成的完整句子是  
“skill 先被扫描和解析，再结合配置判断是否启用，最后被拼进主代理提示词，让模型在运行时遵循这份能力说明。”

### 这个专题里最容易误解的点

你要明确：

- `skill` 不是后端函数
- `skill` 也不是前端组件
- `skill` 更接近“可插拔的任务说明文档”

DeerFlow 的一个特点就是：

“能力不只来自工具，还来自提示词级别的结构化技能说明。”

### 你在这个专题里的阅读顺序

1. 先看 [backend/src/skills/parser.py](/mnt/d/Agent项目/deer-flow/backend/src/skills/parser.py)  
理解一个 `SKILL.md` 是怎么被解析成 Skill 对象的

2. 再看 [backend/src/skills/loader.py](/mnt/d/Agent项目/deer-flow/backend/src/skills/loader.py)  
理解技能是怎么被扫描和加载的

3. 再看 [backend/src/gateway/routers/skills.py](/mnt/d/Agent项目/deer-flow/backend/src/gateway/routers/skills.py)  
理解技能启用/禁用和安装为什么能通过接口管理

4. 最后看 [backend/src/agents/lead_agent/prompt.py](/mnt/d/Agent项目/deer-flow/backend/src/agents/lead_agent/prompt.py)  
理解 skill 怎么进入系统提示词

### 这个专题学完以后你应该能回答

- skill 文件怎么被发现和解析
- skill 怎么进入系统提示词
- skill 的启用/禁用状态存在哪
- skill 和工具到底有什么区别

### 你最终要能自己讲出来的话

“DeerFlow 的 skill 本质上是结构化任务说明，系统会扫描并解析 `SKILL.md`，再把启用中的 skill 注入主代理提示词，让模型按特定工作流做事。”

## 专题 5：subagents

### 目标

搞懂 DeerFlow 是怎么把复杂任务拆给多个子代理的。

### 重点文件

- [backend/src/tools/builtins/task_tool.py](/mnt/d/Agent项目/deer-flow/backend/src/tools/builtins/task_tool.py)
- [backend/src/subagents/executor.py](/mnt/d/Agent项目/deer-flow/backend/src/subagents/executor.py)
- [backend/src/subagents/registry.py](/mnt/d/Agent项目/deer-flow/backend/src/subagents/registry.py)
- [backend/src/subagents/builtins/general_purpose.py](/mnt/d/Agent项目/deer-flow/backend/src/subagents/builtins/general_purpose.py)
- [backend/src/subagents/builtins/bash_agent.py](/mnt/d/Agent项目/deer-flow/backend/src/subagents/builtins/bash_agent.py)

### 这个专题到底在解决什么问题

这个专题解决的是：

“DeerFlow 是怎么把一个复杂任务拆成多个子任务，并让多个子代理去并行完成的？”

这是 DeerFlow 从“带工具的聊天”走向“任务编排系统”的关键点。

### 真实链路

请你用“主代理把一个复杂问题拆给子代理”这个过程去理解：

1. 主代理先收到一个复杂请求  
比如需要同时做调研、读文件、执行命令。  
如果当前模式允许 subagents，主代理会在思考后决定调用 `task(...)`。

2. `task(...)` 不是随便一个函数，而是工具  
打开 [backend/src/tools/builtins/task_tool.py](/mnt/d/Agent项目/deer-flow/backend/src/tools/builtins/task_tool.py)。  
这里就是主代理调用子代理的入口。

3. `task_tool` 先根据 `subagent_type` 选择配置  
它会去找 subagent 注册信息。  
这部分你再看 [backend/src/subagents/registry.py](/mnt/d/Agent项目/deer-flow/backend/src/subagents/registry.py)。

4. 注册表里定义了有哪些内置子代理  
比如：
- `general-purpose`
- `bash`

它们的具体定位与提示词在：
- [backend/src/subagents/builtins/general_purpose.py](/mnt/d/Agent项目/deer-flow/backend/src/subagents/builtins/general_purpose.py)
- [backend/src/subagents/builtins/bash_agent.py](/mnt/d/Agent项目/deer-flow/backend/src/subagents/builtins/bash_agent.py)

5. `task_tool` 不会让子代理从零开始  
它会把父代理的一些必要上下文传下去，比如：
- 当前 thread_id
- sandbox 状态
- thread_data
- 父代理使用的模型信息

这一步很重要，因为它解释了为什么子代理还能继续在同一个 thread 工作区里操作文件。

6. 真正执行发生在 [backend/src/subagents/executor.py](/mnt/d/Agent项目/deer-flow/backend/src/subagents/executor.py)  
这里会创建 `SubagentExecutor`，然后做几件事：
- 过滤子代理能用的工具
- 创建子代理自己的 agent
- 把任务包装成新的输入状态
- 放进后台线程池执行

7. 子代理运行时不是静默的  
`task_tool` 会持续轮询后台结果。  
如果子代理产生了新消息，`task_tool` 会把这些消息作为 `task_running` 事件流式发给前端。

8. 前端为什么能看到子任务进度  
因为前端在 [frontend/src/core/threads/hooks.ts](/mnt/d/Agent项目/deer-flow/frontend/src/core/threads/hooks.ts) 里监听了自定义事件，然后消息列表会把 subtask 卡片渲染出来。

9. 子代理结束后，结果不是直接给用户  
而是先回到主代理。  
主代理再把多个子代理结果整合，最后生成给用户看的最终回答。

10. 你在脑中要形成的完整句子是  
“主代理调用 `task` -> `task_tool` 选择子代理并传递必要上下文 -> executor 在后台启动子代理 -> 进度流式回前端 -> 结果回主代理 -> 主代理统一汇总”

### 这个专题里最重要的三个点

1. `task` 只是入口，不是执行本体  
真正执行在 [backend/src/subagents/executor.py](/mnt/d/Agent项目/deer-flow/backend/src/subagents/executor.py)

2. 子代理不是随便递归套娃  
默认会禁止子代理再继续拥有 `task` 工具，避免无限嵌套

3. 子代理既隔离上下文，又继承必要环境  
它们不是完全独立的陌生进程，而是“带着必要运行环境的独立工作上下文”

### 你在这个专题里的阅读顺序

## 专题 6：memory

### 目标

搞懂 DeerFlow 怎么把跨会话信息保留下来。

### 重点文件

- [backend/src/agents/memory/updater.py](/mnt/d/Agent项目/deer-flow/backend/src/agents/memory/updater.py)
- [backend/src/agents/memory/queue.py](/mnt/d/Agent项目/deer-flow/backend/src/agents/memory/queue.py)
- [backend/src/agents/memory/prompt.py](/mnt/d/Agent项目/deer-flow/backend/src/agents/memory/prompt.py)
- [backend/src/gateway/routers/memory.py](/mnt/d/Agent项目/deer-flow/backend/src/gateway/routers/memory.py)

### 这个专题到底在解决什么问题

这个专题解决的是：

“为什么 DeerFlow 能跨会话记住用户的一些背景、偏好和历史上下文？”

这个模块本质上是在做一件事：

把会话中的长期有效信息抽出来，保存下来，并在之后的对话中重新注入。

### 真实链路

请你用“本次对话结束后，系统如何把长期有价值的信息记住”这个过程去理解：

1. 一次对话执行完后，主代理并不会立刻把聊天记录原样存成 memory  
真正负责触发 memory 更新的是 `MemoryMiddleware`。  
它会把本次会话的消息送进 memory 更新队列。

2. 队列逻辑在 [backend/src/agents/memory/queue.py](/mnt/d/Agent项目/deer-flow/backend/src/agents/memory/queue.py)  
这里最重要的事是 debounce。  
也就是：不是每来一条消息就马上更新，而是先缓一会，防止频繁调用模型。

3. 等到 debounce 时间到了，队列才真正开始处理  
此时会调用 [backend/src/agents/memory/updater.py](/mnt/d/Agent项目/deer-flow/backend/src/agents/memory/updater.py) 里的 `MemoryUpdater`。

4. `MemoryUpdater` 先读取当前 memory 文件  
还是在 [backend/src/agents/memory/updater.py](/mnt/d/Agent项目/deer-flow/backend/src/agents/memory/updater.py)。  
这里会：
- 找到 memory 文件路径
- 读取已有 memory
- 如果文件不存在就创建空结构

5. 然后它会把“旧 memory + 本次对话”一起交给模型  
具体格式要求在 [backend/src/agents/memory/prompt.py](/mnt/d/Agent项目/deer-flow/backend/src/agents/memory/prompt.py)。  
这一步非常关键，因为模型不是自由发挥，而是被要求输出结构化 JSON。

6. 模型返回新的 memory 更新结果后  
`updater.py` 会把这些结果合并进现有 memory，并保存回 JSON 文件。

7. 下一次对话开始时，memory 会重新进入主代理提示词  
也就是说，memory 的作用不是“保存聊天记录”，而是“把长期有价值的信息重新喂给后续对话”。

8. 如果前端想查看 memory  
它不会直接读文件，而是调用 Gateway 的 memory 接口：  
[backend/src/gateway/routers/memory.py](/mnt/d/Agent项目/deer-flow/backend/src/gateway/routers/memory.py)

9. 你在脑中要形成的完整句子是  
“对话结束 -> middleware 把消息送进队列 -> 队列防抖后调用 updater -> updater 读旧 memory 和新对话 -> 模型提取结构化更新 -> 写回 memory 文件 -> 后续对话再注入使用”

### 为什么这个模块不能简单粗暴地做

这里最关键的不是“记住”，而是“什么时候记、记什么、记多少”。

所以系统才需要：

- `prompt.py` 约束 memory 提取格式
- `queue.py` 做 debounce
- `updater.py` 做读写、缓存和更新

否则每条消息都立刻更新，不仅浪费模型调用，还会让 memory 非常混乱。

### 你在这个专题里的阅读顺序

1. 先看 [backend/src/agents/memory/prompt.py](/mnt/d/Agent项目/deer-flow/backend/src/agents/memory/prompt.py)  
理解 memory 更新时要求模型输出什么

2. 再看 [backend/src/agents/memory/queue.py](/mnt/d/Agent项目/deer-flow/backend/src/agents/memory/queue.py)  
理解为什么要排队和防抖

3. 再看 [backend/src/agents/memory/updater.py](/mnt/d/Agent项目/deer-flow/backend/src/agents/memory/updater.py)  
理解 memory 文件如何读取、更新、保存

4. 最后看 [backend/src/gateway/routers/memory.py](/mnt/d/Agent项目/deer-flow/backend/src/gateway/routers/memory.py)  
理解前端如何查看 memory

### 这个专题学完以后你应该能回答

- memory 什么时候被提取
- 为什么要有 debounce 队列
- memory 数据怎么进入后续对话
- 为什么 memory 既不是聊天记录原样保存，也不是简单摘要

### 你最终要能自己讲出来的话

“DeerFlow 会在对话后把长期有价值的信息加入 memory 更新队列，经过防抖后由模型提取并写入持久化存储，再在后续对话中注入系统提示词。”

## 专题 7：MCP 与 community tools

### 目标

搞懂 DeerFlow 怎么扩展外部工具能力。

### 重点文件

- [backend/src/mcp/client.py](/mnt/d/Agent项目/deer-flow/backend/src/mcp/client.py)

- [backend/src/community/jina_ai/tools.py](/mnt/d/Agent项目/deer-flow/backend/src/community/jina_ai/tools.py)

### 这个专题到底在解决什么问题

这个专题解决的是：

“DeerFlow 除了内置工具以外，怎么继续接入外部能力？”

这一层就是 DeerFlow 的扩展能力来源。

### 真实链路

请你用“系统要增加一个外部能力”这个过程去理解：

1. 一种扩展方式是 community tools  
先看 [backend/src/community/tavily/tools.py](/mnt/d/Agent项目/deer-flow/backend/src/community/tavily/tools.py) 和 [backend/src/community/jina_ai/tools.py](/mnt/d/Agent项目/deer-flow/backend/src/community/jina_ai/tools.py)。  
这些工具是项目自己写好的 Python 工具，直接存在于代码仓库里。

2. 另一种扩展方式是 MCP  
MCP 的意思可以先简单理解成：“外部工具服务器把能力暴露出来，DeerFlow 再去连接它”。  
这条链路的起点不是 Python 函数，而是 MCP server 配置。

3. MCP 配置首先由 Gateway 管  
看 [backend/src/gateway/routers/mcp.py](/mnt/d/Agent项目/deer-flow/backend/src/gateway/routers/mcp.py)。  
前端可以通过这个接口读取或修改 MCP server 配置。

4. 真正使用这些配置时，LangGraph 侧会重新读取配置文件  
看 [backend/src/mcp/client.py](/mnt/d/Agent项目/deer-flow/backend/src/mcp/client.py)。  
这个文件负责把配置转换成 MCP 客户端真正需要的参数，比如：
- transport 类型
- command
- args
- url
- headers

5. 然后 [backend/src/mcp/tools.py](/mnt/d/Agent项目/deer-flow/backend/src/mcp/tools.py) 会创建 MCP 客户端并拉取工具列表  
这里的重点不是某个具体类名，而是逻辑：  
“根据配置连接外部 MCP server，再把对方暴露的工具拿回来。”

6. 不管是 community tools 还是 MCP tools，最后都要进入总工具系统  
也就是 [backend/src/tools/tools.py](/mnt/d/Agent项目/deer-flow/backend/src/tools/tools.py)。  
主代理真正可用的工具集，是在这里统一汇总出来的。

7. 所以完整链路应该记成两条：
- community：项目内置 Python 工具 -> 汇总进总工具系统 -> 主代理调用
- MCP：前端改配置 -> Gateway 保存配置 -> LangGraph 读取配置 -> 连接 MCP server -> 拉取工具 -> 汇总进总工具系统 -> 主代理调用

8. 你在脑中要形成的完整句子是  
“community tools 是项目内置工具，MCP tools 是外部 server 暴露的工具；配置由 Gateway 管，真正加载发生在 LangGraph 的工具汇总阶段。”

### 为什么配置在 Gateway，加载在 LangGraph

这是这个专题最关键的结构点。

原因是：

- 前端要修改 MCP 配置，所以需要 HTTP 接口，这就适合放 Gateway
- 真正调用工具的是主代理，所以真正加载工具必须发生在 LangGraph 进程里

也就是说：

“Gateway 负责改配置，LangGraph 负责用配置加载工具。”

### 你在这个专题里的阅读顺序

1. 先看 [backend/src/community/tavily/tools.py](/mnt/d/Agent项目/deer-flow/backend/src/community/tavily/tools.py) 和 [backend/src/community/jina_ai/tools.py](/mnt/d/Agent项目/deer-flow/backend/src/community/jina_ai/tools.py)  
理解一个 community tool 长什么样

2. 再看 [backend/src/mcp/client.py](/mnt/d/Agent项目/deer-flow/backend/src/mcp/client.py)  
理解 MCP server 配置是怎么转换成客户端参数的

3. 再看 [backend/src/mcp/tools.py](/mnt/d/Agent项目/deer-flow/backend/src/mcp/tools.py)  
理解 MCP 工具是怎么被拉进来的

4. 最后看 [backend/src/gateway/routers/mcp.py](/mnt/d/Agent项目/deer-flow/backend/src/gateway/routers/mcp.py)  
理解前端如何通过 Gateway 改 MCP 配置

### 这个专题学完以后你应该能回答

- MCP 工具是怎么被接入系统的
- community tools 和 MCP tools 的区别是什么
- 为什么配置放在 Gateway 而真正工具加载发生在 LangGraph 侧
- DeerFlow 的工具能力为什么能持续扩展
